‍

服务：

  1. api：后端服务

  2. worker：Celery worker

  3. worker-beat：Celery 定时任务

  4. web：前端

  5. db：数据库 postgres

  6. redis：缓存数据库

  7. plugin_daemon：运行插件（Plugin）的执行环境

  8. ssrf_proxy：请求代理，防止 SSRF（Server-Side Request Forgery）攻击

  9. certbot：自动申请和续期 SSL 证书

  10. nginx：Web 反向代理和负载均衡

  11. weaviate：向量数据库（Vector DB）。

  12. sandbox：一个轻量、安全的代码运行时（沙箱）

    开发者的草稿箱，快速调试逻辑；稳定后要迁移到tool、module

    用来在受限环境中执行用户提交的代码（如 Workflow 的 Code 节点、模板/Transform、工具节点的代码解释器等），从而在不危及主机/平台安全的前提下实现自定义逻辑与扩展

    设计目标：

    • 允许用户在 Workflow 中运行自定义脚本（Python/Node 等），以实现数据处理、HTTP 请求、格式转换、简单计算和工具封装等功能。
    • 严格隔离用户代码与宿主系统，限制系统调用、文件/磁盘和网络访问，防止恶意或出错代码破坏平台或窃取数据

    典型用途:

    • Code 节点：在工作流中执行自定义 Python / JS 代码（数据清洗、请求第三方 API、调用内部服务等）。
    • Template Transform / Jinja2：LLM 节点中对模板的渲染或模板中嵌入的小段代码。
    • Tool 节点（Code Interpreter） ：作为工具执行环境，运行短脚本以实现更复杂的 agent 行为

    举例（两种典型例子）:

    1. Workflow 的 Python Code 节点 — 做一个外部 API 请求并返回处理结果

      python
      # Code node (Python)
import requests

def run(input_data):
    # input_data 由 workflow 传入
    url = "https://api.example.com/summary"
    payload = {"text": input_data["content"]}
    resp = requests.post(url, json=payload, timeout=5)
    return {"summary": resp.json().get("summary", "")}

    2. 在 LLM 节点里用 Jinja2 Template Transform 处理数据

      jinja
      {% set user = input.user %}
Hello {{ user.name }} — here are the recent items:
{% for item in input.items %}
- {{ item.title | truncate(60) }}
{% endfor %}
