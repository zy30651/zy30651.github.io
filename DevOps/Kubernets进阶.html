搭建 Kubernetes 集群 ​

  1. 搭建方式 ​

    • minikube：本地开发，单节点集群；学习、本地调试；复杂度最低
    • kubeadm：生产级集群部署，适合中小规模、标准化部署；复杂度中等
    • 二进制安装：完全手动部署，深度定制、大规模生产环境；复杂度最高
    • 命令行工具：特定场景辅助工具，特定云平台或环境；复杂度较低

资源管理 ​

  1. 资源管理介绍

    Kubernetes 中，所有的内容都抽象为资源，用户需要操作资源来管理 kubernetes

    k8s 本质是集群系统，用户在集群中部署各种服务，实际上是在 k8s 内部运行 1 个个容器，并将指定程序跑在容器中

    k8s 最小管理单元不是容器，是Pod；只能将容器放到Pod中，而 k8s 通过Pod 管理器来管理 Pod

    Pod 可以提供服务之后，就要考虑如何访问 Pod 中的服务，k8s 提供了 Service 资源实现这个功能。

    如果 Pod 中程序的数据要持久化，k8s 还提供了各种存储系统

    ReplicationController曾用于动态更新 pod 副本数量，但是因为功能单一，不支持多个 selector 选择，所以慢慢被 ReplicaSet 取代；

    ReplicaSet被广泛用于底层管理 Pod，用户使用封装后的使用封装后的Deployment；

    Deployment通过管理 ReplicaSet 间接管理 Pods，提供滚动更新、回滚功能；

    为什么不直接使用 ReplicaSet？

    1. Deployment 更易用：提供了更高级的功能，滚动更新、回滚；而 ReplicaSet 只负责维持 Pod 的副本数量
    2. 声明式管理：用户只需要定义期望状态，k8s 会自动处理副本的创建、更新、删除
    3. 更好的集成：Deployment 与 Helm/kustomize 等工具集成更好，便于实现复杂部署

    Deployment 关联 ReplicaSet，ReplicaSet 内部嵌套了 pod；

    Deployment是一个更高层次的控制器，通过管理ReplicaSet简洁管理Pods

    当你创建 1 个Deployment 时，k8s 会自动创建 1 个或多个ReplicaSet并由这些ReplicaSet管理 Pod

    ReplicaSet和Pod的关系

    • ReplicaSet通过selector匹配目标 Pods，并确保指定数量的 Pod 副本始终处于运行状态
    • ReplicaSet并不直接嵌套Pods，而是通过PodTemplateSpec定义Pod模板，k8s 根据模板创建和管理Pods

  2. 探针

    容器内应用的监测机制，根据不同的探针来判断容器应用当前的状态

    1. 类型

      1. StartupProbe

        **作用：**启动探针，延迟其他探针的执行，直到容器成功启动。

        作用：仅在启动阶段运行，成功后不再执行

        适用于：应用需要长时间启动的情况

      2. livenessProbe

        **作用：**存活探针，加测容器是否正常运行；

        **行为：**探测失败 kubelet 会重启容器，只有超过设置的次数才会正式启动失败；

        适用于：死锁、无限循环等导致服务不可用但进程仍在运行

        的

      3. ReadinessProbe

        **作用：**就绪探针，监测容器是否准备好接受流量。探测失败，Pod 会从 Service 的 Endpoint 列表中移除。

        **行为：**探测失败时停止流量转发，但不重启容器；

        **适用于：**应用启动时需要初始化配置或依赖服务。

    2. 探测方式

      1. ExecAction

        在容器内执行命令，若命令退出码为 0 则视为成功

        适合自定义监测

        yaml
        startupProbe:
  exec:
    command:
      - sh
      - -c
      - "curl -fs http://localhost:8080/healthz || exit 1"

      2. TcpSocketAction

        尝试与容器指定端口建立 TCP 连接，若连接成功则视为健康

        适合 nginx、redis

        yaml
        readinessProbe:
  tcpSocket:
    port: 3306

      3. HttpGetAction

        发送 HTTP 请求，通过状态码判断是否成功(2xx/3xx 为成功)

        适合微服务

        yaml
        livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    httpHeaders:
      - name: Custom-Header
        value: "value"

    3. 参数配置

      所有探针共享以下参数

      yaml
      livenessProbe:
  initialDelaySeconds: 30 # 容器启动后等待多久开始探测（默认0）
  periodSeconds: 10 # 探测频率（默认10秒）
  timeoutSeconds: 5 # 探测超时时间（默认1秒）
  successThreshold: 1 # 连续成功次数视为探测成功（默认1）
  failureThreshold: 3 # 连续失败次数视为探测失败（默认3）

    4. 生命周期

      Readiness 生命周期和 Liveness 一样贯穿整个 Pod 生命周期

      1. Pod 退出流程->删除操作
        1. Endpoint 删除 pod 的 ip 地址
        2. Pod 变成 Terminating 状态

补充 ​

STS 和 Deploy 在扩容、缩容、更新方面的区别

StatefulSet 直接关联到 Pod

Deployment 关联到 RS，RS 关联 Pod

  1. 扩容、缩容
  2. Deploy：
    1. 无状态应用，可以快速扩容、缩容；所有 Pod 都是相同的
    2. 快速创建副本，副本几乎完全相同，没有特定顺序
    3. 缩容时，随机选择要终止的 Pod；因为无状态，缩容简单且快速
  3. STS：
    1. 有状态应用，有序扩容，按照顺序为每个 Pod 分配一个递增索引(web-0,web-1)；
    2. 新 Pod 会使用预先定义的 PVC 模板来申请持久化存储，确保每个 Pod 都有独立且稳定的存储；
    3. 每个 Pod 都有稳定的网络标识，如 DNS，对于有状态应用数据库很重要
    4. 缩容时，按照逆序终止 Pod(从 web-2->web-1)
    5. 缩容时在确保删除 Pod 前，其数据正确备份和迁移
    6. 缩容后，相关 PVC 仍然保留，除非手动删除，有助于恢复数据
  4. 更新
  5. Deploy
    1. 滚动更新：Deplo 使用滚动更新策略，逐步替换旧的 Pod，确保高可用
    2. 无状态兼容：由于是 Pod 无状态，更新过程碱大， 只需替换容器镜像或配置
    3. 多种更新策略：RollingUpdate、Recreate
  6. STS
    1. 有序滚动更新：STS 也支持滚动更新，但更新过程是有序的，按照索引从高到底
    2. 版本控制：每个 Pod 的更新都可基于索引顺序进行，确保更新过程中的高可用和一致性
    3. 暂停和恢复：支持在更新过程中暂停和恢复，允许用户在发现问题时中止更新并进行修正
    4. 部分更新限制：某些更新操作(更改 Pod 主机名和存储配置)需要谨慎，可能影响那个 Pod 的稳定性和数据持久性
  7. 删除
  8. STS 和 Pod 关联，但是有数据关联
    1. 默认级联删除，删除 Pod
    2. 非级联删除，--cascade=false，不删除 pod
  9. Deploy 删除，会自动级联删除 RS、Pod

DaemonSet ​

为每一个匹配的 Node 节点部署一个守护进程(Pod 副本)。

主要用途： ​
  1. 日志收集：在每个节点上运行日志代理（Fluentd、Filebeat），以收集和转发日志到集中式日志存储系统
  2. 监控和指标收集：部署监控代理（Prometheus Node Exporter、Datadog Agent）在每个节点上收集系统和应用的性能指标
  3. 安全代理：运行安全相关的代理和扫描工具（Falco、Sysdig Secure）以监控和保护每个节点的安全
  4. 网络插件：部署网络插件（Calico、Flannel）的 DaemonSet 来管理每个节点的网络配置和策略
  5. 服务网格：使用服务网格（Istio 的 Sidecar 代理）在每个节点上部署代理，以实现流量管理、安全性、可观察性

Service ​

内部微服务之间的通信

数据库、缓存等后端服务的访问

集群内部组件的服务发现

运行在某一个节点上，通过 endPoints(终端)存储 Service 和 Pod 的 ip 和端口信息，通过 iptables 转发，找到 kube-proxy，通过 kube-proxy 代理具体的容器，以实现交互

Service 的定义 ​
  1. 内部负载均衡：在 k8s 集群内部为一组 Pod 提供稳定的网络端点，实现负载均衡和服务发现
  2. 服务抽象：通过 Service，可以将一组 Pod 抽象成一个统一的网络服务，无论 Pod 的生命周期如何变化（如重启、扩缩容），Service 的 IP 和 DNS 的名称保持不变
  3. 可以跨命名空间
类型 ​

  1. ClusterIP：为 Service 分配 1 个在集群内部唯一的 IP 地址；适合集群内部服务之间通信

  2. NodePort：在每个节点上打开一个固定端口(30000-32767)，通过节点的 IP 和端口与 Service 通信

    适合开发和测试环境，或需要通过节点 IP 直接访问服务的场景，可以暴露给外部，但是实际生产环境不推荐，效率较低，算法 O(n)，节点越多越慢 ，而且 Service 是 4 层负载

  3. LoadBalancer：在云提供商(AWS/GCP)中创建 1 个外部负载均衡器，自动配置将外部流量路由到 Service；适合需要暴露服务给外部访问的生产环境

  4. ExternalName：通过返回 CNAME 记录，将 Service 映射到外部 DNS 名称；适合集成外部服务（外部数据库、API）

  2. 多文件独立处理
  3. 动态路径隔离

使用方法

  1. 定义 volumes 时，需要增加 items 属性，配置 key 和 path，且 path 的值不能从/开始
  2. 在容器内的 volumeMounts 中增加 subPath 属性，该值与 volumes 中的 items.path 的值相同
bash
#下面配置，在spec下的template里边
spec:
	containers:
	- image: nginx:1.7.9
		# 配置加载volume，看加载什么东西，到什么位置
		volumeMounts:
		- name: nginx-conf  # 数据卷名称
# 			mountPath: '/etc/nginx/'  # 挂载路径，
			mountPath: '/etc/nginx/nginx.conf'  # 需要完整路径加文件名
			subPath: etc/nginx/nginx.conf # 与volumes.[0].items.path相同
# 加载configMap数据卷
volumes:
- name:nginx-conf
	configMap:  # 类型为configMap
	  name: nginx-conf-cm # configMap的名字
		items: # 将configMap中那些数据挂载进来
		- key: nginx.conf # 指定挂载那个key
#			path: nginx.conf # 挂载后的key重命名为什么
			path: etc/nginx/nginx.conf # 不以/开头

默认方式：会更新，更新周期是更新时间+缓存时间

subPath：不会更新

变量形式：如果 Pod 中的一个变量从 configMap 或 secret 中得到，同样是不会更新

折中方案：

假设 configMap 中有 nginx.conf 文件；

容器里边/etc/nginx/有很多配置文件；在不希望覆盖的情况下

可以在容器在再创建一个/configmap/nginx/nginx.conf，把 configMap 的 nginx.conf 映射到容器内的/configmap/nginx 下；

在容器启动时，将/etc/nginx/nginx.conf 删除，使用软连接关联/configmap/nginx/nginx.conf 和/etc/nginx/nginx.conf

此处：如果/etc/nginx/nginx.conf 原有文件不删除，无法创建软连接；

删除时机：postStart

PV 与 PVC ​

Volumes 是依赖于本机实现；

NFS 挂载是依赖于远程服务实现

PV 和 PVC 是更高层次的封装

PV：

  1. 持久卷，是集群中的 1 块存储(磁盘/NFS/云存储)，是集群资源，像节点一样；与普通 Volume 一样，也是使用卷插件来实现，只是他们有独立于 Pod 的生命周期。即使 Pod 被删除 PV 也会保留；

  2. 抽象于底层细节(磁盘类型、存储系统)，供 Pod 通过 PVC 申请使用

PVC：

  1. 持久卷申领，表达的是用户对存储的请求，概念与 Pod 类似；Pod 会消耗节点资源，PVC 会消耗 PV 的资源。Pod 可以请求特定数量的资源(CPU 和内存)，PVC 可以申领特定的大小和访问模式。
  2. k8s 根据 PVC 自动绑定到符合条件的 PV；用于解耦应用和存储细节；用户只需声明需求，

CSI：

  1. 容器存储接口，一个标准化插件接口，允许第三方存储系统集成到 k8s；
  2. 扩展存储支持更多后端、动态卷供应可以按需创建 PV、高级功能支持如卷快照、克隆、扩容等
  3. AWS EBS、Google Persistent Disk、NFS、Ceph RBD

  1. 生命周期

    1. 构建

      1. 静态构建：手动创建 PV
      2. 动态构建：创建StorageClass（引用 CSI 驱动)

    2. 使用

      1. 用户创建 PVC，指定所需存储大小和访问模式

    3. 绑定

      1. 静态模式：将 PVC 绑定到现有 PV；如果找不到 PV 会查看StorageClass，有则转为动态，否则一直处于未绑定状态，Pod 无法启动；因为需要等待绑定成功才能启动
      2. 动态模式：通过 CIS 驱动自动创建 PV 并绑定到 PVC

    4. Pod 使用

      1. Pod 挂在 PVC，应用数据写入持久化存储

    5. 回收策略

      用户不再使用存储卷时，可以从 API 中将 PVC 对象删除，从而允许资源被回收再利用。

      1. 保留：删除 PVC 时，PV 卷仍存在，都应数据卷被视为已释放；卷仍然存折前 PVC 的数据，不能用于其他申领。可以手动回收
        1. 删除 PV 对象，位于外部基础设置的存储资产仍存在
        2. 根据情况，手动清除锁关联的存储资产上的数据
        3. 手动删除所关联的存储资产
      2. 删除：对于支持 Delete 策略的卷插件，删除策略会将 PV 对象从 k8s 中移除，同时也会从外部基础设置中移除。动态制备的卷会集成StorageClass的策略，默认值为 Delete；管理员需要根据用户期望来配置 StorageClass，否则 PV 卷被创建之后必须要被编辑或修补
      3. 回收：已废弃，取而代之的是：使用动态制备；如果下层的卷插件支持，回收策略会在卷上执行基本的擦出(rm -rf /thevolume/)操作，之后允许该卷用于新的 PVC 申领

  2. PV

    1. 状态

      1. Availabel：空闲，未被绑定
      2. Bound：已被 PVC 绑定
      3. Released：PVC 被删除，资源已回收，但 PV 未被重新使用
      4. Failed：自动回收失败

    2. 配置文件

污点和容忍 ​

污点（Taints） ​

是节点的属性，用于标记节点具有某种缺点，阻止不容忍这些污点的 Pod 被调度到该节点

对应驱逐策略：

  • NoSchedule：仅禁止新 Pod 调度到该节点
  • PreferNoSchedule：尽量避免调度，但非强制
  • NoExecute：不仅禁止新 Pod，还会驱逐已运行但未容忍的旧 Pod

操作指令：

bash
# 添加污点
kubectl taint nodes <node-name> key=value:NoSchedule
kubectl taint nodes k8s-node1 memory=low:NoSchedule
# 基于角色创建污点
kubectl taint nodes k8s-node1 node-role.kubernetes.io/master:NoSchedule
# 示例：标记节点为仅GPU任务使用
kubectl taint nodes gpu-node1 hardware=gpu:NoSchedule
# 删除污点
kubectl taint nodes <node-name> memory=low:NoSchedule-
容忍（Tolerations） ​

是 Pod 的属性，允许调度到具有特定污点的节点，但不会强制调度（需与节点亲和性结合强制）

通过tolerations字段配置，支持精确匹配、通配符、容忍所有污点

Equal：必须与污点值做匹配，key/value 都必须相同，才能够容忍该节点

Exists：只比较 key，不比较 value；

yaml
spec:
	containers:
	- name: main
		image: nvidia/cuda-app
		tolerations:
		- key : "memory" # 匹配污点的键
			operator : "Equal" # 操作符（Equal/Exists）
			value : "low" # 匹配值（operator=Equal时必须）
			effect : "NoSchedule" # 污点效果（可忽略则容忍全部效果）
			tolerationSeconds: 10

亲和力（Affinities） ​

分类和作用

  • 节点亲和力：让 Pod 倾向调度到满足条件的节点

    • requiredDuringSchedulingIgnoredDuringExecution：硬性满足

    • preferredDuringSchedulingIgnoredDuringExecution：软性满足

    • 应用：

      • 匹配类型

        • In：满足 1 个
        • NotIn：一个都不满足，反亲和性
        • Exists：只要存在
        • DoesNotExist：只有不存在
        • Gt：必须要大于节点数值
        • Lt：小于节点数值

      • 配置模板

        yaml

  • Pod 亲和力：控制 Pod 与其他 Pod 的部署关系

    • 亲和力：吸引 Pod 与其他特定 Pod 共置（如提升缓存命中）

      将与指定 Pod 亲和性相匹配的 pod 部署在同一节点

      • RequiredDuringSchedulingIgnoredDuringExecution
      • RequiredDuringSchedulingIgnoredDuringExecution

    • 反亲和力：避免 Pod 与其他 Pod 共置（如提高可用性）

      根据策略尽量部署或不部署到一起

      • RequiredDuringSchedulingIgnoredDuringExecution
      • RequiredDuringSchedulingIgnoredDuringExecution

实例：

  1. 节点亲和力（硬性要求）

    yaml
    spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                  - us-west1-a

  2. Pod 反亲和力（避免同服务部署在同一节点）

    yaml
    spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values:
                  - web
          topologyKey: "kubernetes.io/hostname"

三者协同作用

机制作用方向使用场景
污点与容忍节点 →Pod专用节点管理（如 GPU 节点、不可用节点标记）
节点亲和力Pod→ 节点优化资源分配（如优先高性能节点、特定区域部署）
Pod 亲和/反亲和Pod↔Pod提升服务性能（亲和）或确保高可用（反亲和）

实际场景 ​

场景 1：专用 GPU 节点 ​

确保只有声明了 GPU 需求的 Pod 才能调度到标记为 GPU 专用节点，并优先选择此类节点。

  1. 节点打污点

    阻止未容忍此污点的 Pod 调度到该节点。

    当 Pod 配置了对应污点的 tolerations，调度器会将其加入候选列表，但 不强制必须调度到该节点。

    bash
    kubectl taint nodes gpu-node hardware=GPU:NoSchedule

  2. Pod 配置容忍和节点亲和

    preferredDuringSchedulingIgnoredDuringExecution 表示软性偏好，调度器会尝试将 Pod 分配到 hardware=GPU 的节点，但若无可用节点仍可调度到其他节点。 weight: 100 给予此条件最高优先级（相对于其他亲和性规则）

    yaml
    spec:
  # 容忍配置：允许 Pod 接受 GPU 节点的污点
  tolerations:
    - key: "hardware" # 匹配污点的键
      operator: "Equal" # 精确匹配值（若为 Exists 则忽略值）
      value: "GPU" # 污点值必须为 GPU
      effect: "NoSchedule" # 仅容忍 effect 为 NoSchedule 的污点

  # 节点亲和性：引导调度器优先选择 GPU 节点
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100 # 优先级权重（范围 1-100）
          preference:
            matchExpressions:
              - key: hardware # 节点标签的键
                operator: In # 匹配方式（In/NotIn/Exists 等）
                values: ["GPU"] # 节点标签值需在列表中
场景 2:Web 服务高可用 ​

避免同一服务的多个 Pod 部署到同一物理节点，减少单点故障风险。

requiredDuringSchedulingIgnoredDuringExecution 硬性要求，调度期间必须满足，运行时若条件不满足（如节点故障）也不会驱逐已运行的 Pod。

topologyKey: "kubernetes.io/hostname" 定义拓扑域的粒度，此处按节点划分，确保同一服务的 Pod 不在同一节点。

Web 服务 Pod 需带有app: web标签

yaml
metadata:
  labels:
    app: web

调度器决策：

  1. 当新 Pod 被创建时，调度器检查各节点的拓扑域（此处为节点名称）。
  2. 如果某节点已有带 app=web 标签的 Pod，则排除该节点。

部署结果 假设集群有 3 个节点，Web 服务副本数为 3，则每个节点部署 1 个 Pod。

跨可用区部署 ： 设置 topologyKey: topology.kubernetes.io/zone ，确保 Pod 分布在不同的可用区。

软性反亲和 ： 使用 preferredDuringSchedulingIgnoredDuringExecution 尽量分散 Pod，但不强制

yaml
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app # 匹配 Pod 的标签键
                operator: In # 标签值需在指定列表中
                values: ["web"] # 目标服务标签值
          topologyKey: "kubernetes.io/hostname" # 拓扑域为节点级别

身份认证和权限 ​

认证 ​

验证用户或服务的身份

User Accounts ​

普通账户：是假定被外部或独立服务管理的。由管理员分配 keys，用户像使用 keystone 或 google 帐号一样，被存储在包含 usernames 和 password 的 list 的文件里

k8s 不能通过 api 调用将普通用户添加到集群中

普通用户是针对用户(人)，服务账户是针对 Pod

Service Accounts ​

控制器有 3 种

  1. Service Account Admission Controller：

    自动注入 ServiceAccount 的认证信息(令牌和 CA 证书)

    通过AdmissionController 插件实现对 pod 修改，它是 apiserver 的一部分，创建或更新 pod 时会进行修改 pod。

    当插件处于激活状态(默认)时，修改 pod 会执行下面操作：

    1. 如果 Pod 没有设置 ServiceAccount，则将 ServiceAccount 设置为 default
    2. 确保 pod 引用的 ServiceAccount 存在，否则将会拒绝请求
    3. 如果 pod 不包含任何 ImagepullSecrets 则将 ServiceAccount 的 ImagePullSecrets 会添加到 Pod 中
    4. 为包含 API 访问的 Token 的 Pod 添加了 1 个 volume
    5. 把 volumeSource 添加到安装在 pod 的每个容器中，挂在在/var/run/secrets/kubenetes.io/serviceaccount

  2. Token Controller：

    负责生成和管理 Service Account 的 Secret（即令牌），并与 API 通信。

  3. Service Account Controller：

    负责管理 ServiceAccount 的创建和生命周期

    在 namespace 中管理 ServiceAccount，并确保每个有效的 namespace 都存在一个default的 ServiceAccount

授权 ​

决定用户/服务是否有权限执行操作(RBAC、ABAC)

RBAC：Role-Based Access Control

为什么是基于角色而不是基于权限；因为多个用户可能具有相同权限。配置起来很复杂

Role ​

定义某个命名空间-namspace 内的权限

比如允许读取 Pod

bash
kubectl get role -n ingress-nginx
kubectl get role -n ingress--nginx -oyaml
ClusterRole ​

定义集群-cluster 范围的权限

比如管理节点

bash
kubectl get clusterroles
kubectl get clusterroles ingress-nginx -oyaml
RoleBinding ​

将 Role 绑定到用户/组/Service Account

ClusterRoleBinding ​

将 ClusterRole 绑定到全局实体

准入控制 ​

拦截请求进行额外验证或修改（例如 Pod 的安全策略）
